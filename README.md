# Мошенник.NET (Cyber Shield)

Учебный одностраничный сайт‑тренажёр по кибергигиене: помогает распознавать мошеннические сценарии в диалогах и «под давлением» тренировать безопасные формулировки. Проект специально сделан как статический фронтенд (чтобы легко публиковать на GitHub Pages) и опционально подключаемый прокси‑бекенд (Cloudflare Worker), который даёт «живой» диалог через LLM и при этом не раскрывает API‑ключ в браузере.

Ниже — разбор архитектуры и назначение каждого файла/модуля, чтобы описание можно было использовать в конкурсной работе.

---

## 1) Что делает сайт (функциональные модули)

Сайт состоит из четырёх пользовательских модулей, которые логически связаны общей целью — научить «разрывать» мошенническую схему до момента передачи денег/данных:

1. **«База распознавания / сигналы риска»**  
   Карточки‑маркеры: срочность, запрос кода/пароля, перевод в сторонний канал, «слишком выгодно», подмена контакта, предоплата за доступ к работе.  
   Цель: дать короткие паттерны узнавания, которые можно быстро вспомнить.

2. **«Тренажёр диалога (чат)»**  
   Пользователь отвечает на реплики «собеседника», а система выдаёт:
   - «красные флаги» текущей ситуации,
   - разбор, почему это похоже на мошенничество,
   - оценку реакции пользователя (безопасно/рискованно/неочевидно),
   - при необходимости — «учебный код» (имитация того, что обычно выманивают).

   Реализовано 2 режима:
   - **Онлайн‑режим** (через Cloudflare Worker): ответы генерирует LLM, Worker возвращает структурированный JSON.
   - **Оффлайн‑режим** (без Worker): UI остаётся работоспособным, но без «живых» ответов; часть оценки делается локально эвристиками (регэксп‑паттерны).

3. **«Сценарии/кейсы (ветвящиеся тесты)»**  
   Набор заранее описанных ситуаций, каждая разбита на этапы (`stages`). На каждом этапе пользователь выбирает один из вариантов ответа. Выбор сразу получает:
   - оценку (safe/risk),
   - объяснение,
   - совет (coach tip),
   - при рискованном исходе — «тег ошибки» (используется в прогрессе).

   Важный UX‑элемент: сценарии **открываются последовательно**. Пока не завершён текущий тест, следующий отображается как заблокированный.

4. **«Прогресс»**  
   Метрики, которые сохраняются *локально*:
   - сколько сценариев пройдено,
   - сколько чатов‑сессий проведено,
   - «карта ошибок» — топ повторяющихся mistake‑тегов,
   - дата последней активности.

## 2) Архитектура: почему выбран именно такой стек

### Фронтенд: React + Vite + TypeScript

- **React** — удобен для UI, где много состояний: ветвящиеся сценарии, модалки, вкладки, чат‑лента и «панель коуча».  
- **TypeScript** — фиксирует контракт данных (сценарии, сообщения, прогресс, API‑ответы) и снижает риск «разъезда» фронта и Worker.
- **Vite** — быстрый dev‑сервер и сборка в статический `dist/`, что идеально для GitHub Pages.

### UI‑компоненты: Radix UI Dialog

Используется только модальный компонент (`@radix-ui/react-dialog`), потому что он:
- доступный (a11y) по умолчанию,
- корректно управляет фокусом/оверлеем,
- не навязывает визуальный стиль (стилизация полностью через CSS проекта).

### Данные и состояние: localStorage

Дизайн намеренно «без бекенда»: прогресс хранится **только у пользователя** в браузере. Это:
- упрощает развёртывание,
- избегает хранения персональных данных на сервере,
- делает проект удобным для учебной демонстрации.

### Бекенд (опционально): Cloudflare Worker как прокси к LLM

Worker нужен не для базы данных, а как тонкий безопасный слой:
- **скрывает API‑ключ** (в браузер он не попадает),
- ограничивает CORS по origin,
- валидирует вход/выход (строгий JSON‑контракт),
- подрезает историю диалога, чтобы контролировать длину контекста,
- возвращает устойчивые ошибки, чтобы UI показывал понятные сообщения.

LLM‑провайдер в Worker — **Groq Chat Completions API** (совместимый формат `openai/v1/chat/completions`): низкая задержка и простой контракт.

### Хостинг: GitHub Pages + GitHub Actions

Фронтенд — статические файлы, поэтому публикация делается workflow‑ом: build → upload `dist/` → deploy to Pages.

## 3) Потоки данных (data flows)

### 3.1. Сценарии (локально)

Источник данных: `src/data/content.ts`.  
UI:
- список сценариев (доступность/блокировка),
- модальное окно с таймлайном сообщений и выбором,
- подсчёт «процента безопасных решений»,
- запись тегов ошибок в прогресс.

### 3.2. Чат (локальная логика + опциональный Worker)

1) Пользователь вводит сообщение → фронтенд формирует `ChatApiRequest` и отправляет на `VITE_CHAT_API_URL`.  
2) Worker вызывает LLM и возвращает `ChatApiResponse` (JSON).  
3) Фронтенд:
- добавляет реплику «собеседника» в чат‑ленту,
- обновляет панель коуча,
- увеличивает счётчик сессий,
- при необходимости фиксирует тег ошибки в статистике.

Подстраховка: фронтенд имеет локальную эвристику `evaluateReplyLocally()`, которая на типовых формулировках может переопределить `userWasSafe/mistakeTag`.

## 4) Контракты данных (типизация)

Типы описаны в `src/types.ts` — это центральная схема данных проекта.

Ключевые сущности:
- `Scenario` → `ScenarioStage` → `ScenarioChoice` — структура ветвящегося теста.
- `ProgressState` — то, что сохраняется в `localStorage`.
- `ChatMessage` — элемент истории чата.
- `ChatApiRequest` / `ChatApiResponse` — контракт фронта с Worker.

Смысл: приложение построено на контенте и состоянии; типизация делает его расширяемым (новые сценарии добавляются без правок UI‑логики).

## 5) Реализация фронтенда (модули и ответственность)

### `src/main.tsx` — точка входа React

- подключает `src/index.css`,
- монтирует `<App />` в `#root`.

### `src/App.tsx` — основной модуль приложения (весь UI и логика)

Файл сделан монолитным (один главный компонент), чтобы репозиторий оставался компактным и легко читаемым для учебной/конкурсной работы.

Внутри:

1) **Состояния**
   - `progress` (из `localStorage`),
   - `scenarioModal*` (модальный тест),
   - `chat*` (лента, инпут, параметры, ошибки),
   - `mainTab/chatTab` (мобайл‑вкладки).

2) **Сохранение прогресса**
   - `useEffect` сохраняет `progress` при изменениях.

3) **Мобильная адаптация**
   - `useMediaQuery('(max-width: 719px)')` переключает режимы:
     - desktop: секции на одной странице,
     - mobile: tabbar + отдельные tab‑страницы.
   - для мобайла используется `window.location.hash` (навигация по вкладкам).

4) **Сценарии**
   - определение «текущего» сценария (первый непройденный),
   - `openScenarioModal()` — старт теста,
   - `handleChoice()` — выбор варианта, фиксация результата и обновление прогресса,
   - блокировка будущих сценариев до прохождения текущего.

5) **Чат**
   - `handleChatSubmit()`:
     - валидация input,
     - запрос на Worker,
     - проверка схемы ответа (`isChatApiResponse`),
     - «склейка» оценки с локальной эвристикой,
     - обновление ленты и прогресса.
   - `resetChat()` — новая сессия (новый `sessionId`) и очистка истории.

6) **UI‑композиция**
   - Desktop: `#signals`, `#chat`, `#cases`, `#tips`, `#progress`.
   - Модалка тестов: `@radix-ui/react-dialog`, таймлайн + карточки обратной связи.

### `src/data/content.ts` — учебный контент как данные

Содержит:
- `recognitionCards` — карточки «сигналов риска»,
- `safetyQuotes` — короткие советы,
- `scenarios` — ветвящиеся тесты (этапы/выборы/объяснения/теги ошибок),
- `scenarioHints` и `chatDifficulties` — параметры для чата.

Почему так: контент отделён от UI, поэтому сценарии легко расширять и описывать как «учебные модули» в конкурсной работе.

### `src/lib/storage.ts` — `localStorage` + миграция ключей

- `loadProgress()` читает прогресс и валидирует структуру,
- предусмотрена миграция:
  - v1: `cyber-shield-progress-v1`
  - v2: `moshennik-net-progress-v2`
- `saveProgress()` пишет прогресс.

### `src/types.ts` — доменная схема данных

См. раздел 4.

## 6) Worker (прокси к LLM)

Worker в `worker/` разворачивается отдельно от фронтенда.

### Эндпоинты

- `GET /health` → `{ ok: true }`
- `POST /api/chat` → JSON в формате `ChatApiResponse`

### Что делает `worker/src/index.ts`

- CORS: разрешает только `ALLOWED_ORIGIN` (+ localhost для dev), обрабатывает preflight.
- Валидация payload и «санитария» истории: фильтрация/обрезка.
- Вызов Groq API (`/openai/v1/chat/completions`) с system+user промптами.
- Парсинг результата: `extractJson()` (в т.ч. из fenced blocks) + `normalizeModelResponse()`.
- Эвристика оценки (`evaluateUserReplyHeuristically`) для стабилизации `userWasSafe/mistakeTag`.
- Генерация `simulatedCode` (6 цифр), если «мошенник» просит код.

### `worker/wrangler.toml` и переменные

- `ALLOWED_ORIGIN` — допустимый origin фронтенда,
- `GROQ_MODEL` — модель,
- `GROQ_API_KEY` — секрет (задаётся через `worker/.dev.vars` или secrets в Cloudflare).

## 10) Приватность и безопасность (как задумано)

- Прогресс хранится локально (`localStorage`).
- Worker не хранит данные и не ведёт сессии, он только проксирует запрос в LLM.
- На Worker включены:
  - ограничение origin (CORS),
  - ограничения длины данных,
  - обрезка истории.

В онлайн‑режиме чат‑история уходит в LLM‑провайдер (Groq), поэтому UI предупреждает: не вводить реальные коды/персональные данные.

## 11) Наблюдения и потенциальные улучшения (кратко, по делу)

1) **Кодировка текста в отдельных файлах**  
   В `index.html` заголовок страницы отображается как «моджибейк» (`Р…`). Аналогичные «ломаные» русские строки присутствуют в `worker/src/index.ts` (system prompt/паттерны). Это похоже на несоответствие кодировки исходников ожидаемому UTF‑8 и может ухудшать качество промпта и работу эвристик в Worker.

2) **Выделение компонентов из `src/App.tsx`**  
   Сейчас монолит удобен для учебного репозитория, но при росте логично вынести чат/модалку/секции в отдельные компоненты и хуки.

3) **Тесты на ключевую логику**  
   Можно добавить unit‑тесты на:
   - миграцию/валидацию `storage.ts`,
   - локальные эвристики оценки,
   - нормализацию ответа Worker.
